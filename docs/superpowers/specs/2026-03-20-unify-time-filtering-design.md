# Unify Time Filtering Design

**Date**: 2026-03-20
**Status**: Approved
**Scope**: Refactor all time-related fields and search filtering to use a unified numeric timestamp approach with time period support.

## Problem

1. **Two incompatible time filtering mechanisms**: `event_time_ts` (numeric range) for semantic search vs `time_bucket` (string comparison) for hierarchy search.
2. **`time_bucket` format varies by level**: L0=`YYYY-MM-DDTHH:MM:SS`, L1=`YYYY-MM-DD`, L2=`YYYY-Wxx`, L3=`YYYY-MM`. The search API converts query time ranges to `YYYY-MM-DD` uniformly, so L2/L3 summaries can never be found via time-filtered hierarchy search.
3. **No way to represent time periods**: Summaries and uploaded narrative content cover a time range, but only a single point in time (`event_time`) is stored.
4. **Redundant/confusing fields**: `created_at_ts` (storage time) vs `create_time_ts` (context creation time) cause semantic confusion.

## Solution

### Approach: Clean refactor (Approach A)

One-shot refactor of all time fields and search logic. No backward compatibility with old data.

## Model Changes

### ContextProperties (`opencontext/models/context.py`)

**Delete:**
- `time_bucket: Optional[str]`

**Rename:**
- `event_time: datetime` → `event_time_start: datetime`

**Add:**
- `event_time_end: datetime`

**Rules:**
- Point events (L0), knowledge, documents: `event_time_end = event_time_start`
- Time periods (L1/L2/L3, uploaded summaries): `event_time_start` = period start, `event_time_end` = period end

**Auto-generated by storage layer:**
- `event_time_start_ts` (numeric)
- `event_time_end_ts` (numeric)

### `get_llm_context_string()` method

- `event time: {event_time}` → `event time: {event_time_start}` (+ `~ {event_time_end}` if different from start)
- Delete `time_bucket` line

### ProcessedContextModel (API response model)

- `event_time` → `event_time_start`
- Add `event_time_end`
- Delete `time_bucket`

### Fields unchanged

- `create_time` / `update_time` / `last_call_time` (and their `_ts` derivatives)
- `hierarchy_level`, `refs`

## Search/Retrieval Changes

### Unified time filter: numeric range overlap

All search paths use one filtering approach:

```
event_time_start_ts <= query_end AND event_time_end_ts >= query_start
```

### EventSearchService (`server/search/event_search_service.py`)

- `_build_filters`: Convert `time_range` to overlap condition (two `_ts` fields with range filters). Replaces single-field `event_time_ts` filter.
- `filter_search`: When `hierarchy_levels` present, use same numeric overlap filter instead of `_time_range_to_buckets`.
- Delete `_time_range_to_buckets` method.
- `semantic_search`: Follows `_build_filters` change automatically.

### search_by_hierarchy (both backends)

- Signature: `time_bucket_start/end: str` → `time_start/end: Optional[float]` (timestamps)
- Filter: Numeric range on `event_time_start_ts` / `event_time_end_ts` (VikingDB `range` op, Qdrant `models.Range`)
- Delete in-code string comparison workaround.
- VikingDB: Replace `FIELD_CREATED_AT_TS` sort field with `FIELD_CREATE_TIME_TS` in `search_by_hierarchy` and `get_all_processed_contexts`.

### EventNode (`server/search/models.py`)

- Delete `time_bucket` field
- `event_time` → `event_time_start`
- Add `event_time_end`

### Search route (`server/routes/search.py`)

- Tree sorting: `time_bucket` → sort by `event_time_start`
- EventNode construction: use `event_time_start` / `event_time_end`

### HierarchicalEventTool (`tools/retrieval_tools/hierarchical_event_tool.py`)

- Delete `_ts_to_day_bucket`, `_ts_to_week_bucket`, `_ts_to_month_bucket`
- `_search_summaries` passes numeric timestamps directly
- `_format_result`: `event_time` → `event_time_start`, add `event_time_end`, delete `time_bucket`
- Uniform time range handling across all levels

### BaseContextRetrievalTool (`tools/retrieval_tools/base_context_retrieval_tool.py`)

- `time_type` default: `"event_time_ts"` → `"event_time_start_ts"`
- Parameter schema: update valid sort options
- Output format: `event_time` → `event_time_start`, add `event_time_end`, delete `time_bucket`

### MemoryCacheManager (`server/cache/memory_cache_manager.py`)

- `created_at_ts` → `create_time_ts`
- `event_time_ts` → `event_time_start_ts`
- Daily summary items: `time_bucket` → derive display label from `event_time_start` date portion
- Sorting: `time_bucket` → sort by `event_time_start`

### Memory cache models (`server/cache/models.py`)

- `DailySummaryItem.time_bucket` → `event_time_start` (str, date portion)
- `SimpleDailySummary.time_bucket` → `event_time_start` (str, date portion)

### Monitoring route (`server/routes/monitoring.py`)

- Replace `time_bucket` references with `event_time_start`

## Write Path Changes

### TextChatProcessor (`context_processing/processor/text_chat_processor.py`)

- `event_time` → `event_time_start` (source unchanged: LLM extraction or fallback to `create_time`)
- `event_time_end = event_time_start` (point events)
- Delete `time_bucket` assignment

### AgentMemoryProcessor (`context_processing/processor/agent_memory_processor.py`)

- `event_time` → `event_time_start`
- `event_time_end = event_time_start` (point events)
- Delete `time_bucket` assignment
- Display formatting: replace `time_bucket` usage with `event_time_start` date formatting

### push_base_events (`server/routes/agents.py`)

- `event_time` → `event_time_start`
- `event_time_end = event_time_start` (L0 point events)
- Delete `time_bucket` assignment

### HierarchySummaryTask (`periodic_task/hierarchy_summary.py`)

In `_store_summary`:
- L1 daily: `event_time_start` = day 00:00:00 UTC, `event_time_end` = day 23:59:59 UTC
- L2 weekly: `event_time_start` = Monday 00:00:00, `event_time_end` = Sunday 23:59:59
- L3 monthly: `event_time_start` = first day 00:00:00, `event_time_end` = last day 23:59:59
- Delete `time_bucket` parameter and `_parse_event_time_from_bucket` helper
- Callers (`_backfill_daily_summaries`, `_generate_weekly_summary`, `_generate_monthly_summary`) compute and pass `event_time_start`/`event_time_end` directly
- Formatting methods (`_format_weekly_hierarchical`, `_format_monthly_hierarchical`): derive display labels from `event_time_start`/`event_time_end` instead of `time_bucket`

### DocumentProcessor (`context_processing/processor/document_processor.py`)

- Rename `event_time` → `event_time_start`, `event_time_end = event_time_start`

### Context merger (`context_processing/merger/context_merger.py`, `merge_strategies.py`)

- `event_time` → `event_time_start` in merge copy logic
- Add `event_time_end` copy

- Replace `FIELD_CREATED_AT_TS` sort field with `FIELD_CREATE_TIME_TS` in all query methods
